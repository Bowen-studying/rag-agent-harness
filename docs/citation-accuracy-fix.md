# 迭代1详录（历史方案）：引用准确率 90% → 100%

> 历史说明：这是第一轮中间修复记录。随后发现“最多两条证据”会损害多证据问题的召回率，当前实现已经改为动态证据数量、Token预算和Precision/Recall/F1。后续纠错见 [`engineering-log.md`](engineering-log.md)。

这份记录展示如何从评测报告发现问题、定位根因、选择修复方案、增加可观测性，并用回归测试证明修复没有破坏多来源回答。

## 1. 修复前基线

10 条固定用例的初始结果：

| 指标 | 修复前 |
|---|---:|
| 成功率 | 100% |
| 检索命中率 | 100% |
| 关键词通过率 | 100% |
| 引用准确率 | 90% |
| 自动测试 | 8 项 |

这不是“一道题完全答错”，而是两道题在正确引用之外夹带了无关来源。

| 用例 | 第一条（正确） | 第二条（错误） | 单题引用准确率 |
|---|---|---|---:|
| `case_03` 发布时间窗口 | `release_policy.md#p2` | `security_policy.md#p2` | 50% |
| `case_07` 网关请求限制 | `api_limits.md#p2` | `incident_response.md#p2` | 50% |

其余 8 条为 100%，所以总体为：

```text
(8 × 1.0 + 2 × 0.5) / 10 = 0.9
```

## 2. 根因定位

回答器原来固定取前两条结果：

```python
selected = results[:2]
```

检索器根据词语重合打分，只要得分大于零就可能进入候选。两个错误来源分别被通用词误召回：

- `case_03`：安全策略中的“生产密钥”和问题中的“生产发布”共享“生产”；
- `case_07`：事故响应中的“5 分钟/15 分钟”和问题中的“每分钟”共享“分钟”。

分数进一步说明第二条证据明显弱于第一条：

| 用例 | 第一名得分 | 第二名得分 | 第二名/第一名 | 是否真正需要第二来源 |
|---|---:|---:|---:|---|
| `case_03` | 13.260792 | 5.786460 | 43.64% | 否 |
| `case_07` | 24.796099 | 6.182323 | 24.93% | 否 |
| `case_05` | 27.113401 | 13.707899 | 50.56% | 是 |

`case_05` 是重要的保护用例：“Agent 是否可以直接修改生产数据库？”确实需要同时引用事故规范和安全策略。因此不能简单地永远只取第一条。

## 3. 方案比较

| 方案 | 优点 | 问题 | 结论 |
|---|---|---|---|
| 永远只保留第一条 | 最简单，能消除额外引用 | 会破坏真正的多来源回答 | 放弃 |
| 只保留同一文件的段落 | 可减少跨文档误引 | `case_05` 必须跨文档 | 放弃 |
| 固定绝对分数阈值 | 容易实现 | 不同问题的分数尺度不同 | 放弃 |
| 相对第一名的分数阈值 | 随问题尺度变化，可保留强第二证据 | 阈值仍需更多数据验证 | 采用 |

最终规则：

```text
始终保留第一条证据；
第二条得分 >= 第一条得分 × 50% 时，才允许进入回答；
最多使用两条证据。
```

核心实现：

```python
threshold = results[0].score * 0.5
selected = [results[0]]
for item in results[1:]:
    if len(selected) >= 2 or item.score < threshold:
        break
    selected.append(item)
```

## 4. 增加可观测性

修复同时新增 `evidence_selected` 轨迹事件，记录阈值、保留证据和拒绝证据。以 `case_03` 为例：

```json
{
  "event": "evidence_selected",
  "top_score": 13.260792,
  "min_secondary_score": 6.630396,
  "selected": [
    {"chunk_id": "release_policy.md#p2", "score": 13.260792}
  ],
  "rejected": [
    {"chunk_id": "security_policy.md#p2", "score": 5.78646},
    {"chunk_id": "incident_response.md#p4", "score": 3.351375}
  ]
}
```

因此不只知道“结果变好了”，也能看到某条证据为什么被拒绝。

## 5. 回归测试

新增两类测试：

1. `case_03` 和 `case_07` 的弱第二来源必须被过滤；
2. `case_05` 的两个强来源必须同时保留。

完整评测测试还明确断言：

```python
self.assertEqual(report["summary"]["citation_accuracy"], 1.0)
```

运行：

```bash
python3 -m unittest discover -s tests -v
python3 -m rag_harness.cli eval --cases eval_cases.json --docs sample_docs --output artifacts/eval_report.local.json --trace-dir artifacts/traces
```

## 6. 修复后结果

| 指标 | 修复前 | 修复后 | 变化 |
|---|---:|---:|---:|
| 成功率 | 100% | 100% | 保持 |
| 检索命中率 | 100% | 100% | 保持 |
| 关键词通过率 | 100% | 100% | 保持 |
| 引用准确率 | 90% | 100% | +10 个百分点 |
| 自动测试 | 8 | 10 | +2 |
| 近似输入 Token | 1,019 | 640 | -37.2% |
| 近似输出 Token | 873 | 494 | -43.4% |

Token 减少是因为无关证据不再进入回答。延迟会受运行环境影响，因此没有把几毫秒级变化解释为性能提升。

## 7. 当前边界

100% 只代表当前 10 条虚构用例，不代表生产系统已经达到 100%。50% 相对阈值是在当前小型数据集上得到的可解释基线，后续还需要：

- 增加未参与阈值选择的盲测集；
- 增加共享大量关键词但语义不同的困难负样本；
- 为多跳问题增加更多真实跨文档用例；
- 比较固定阈值、语义重排和 LLM Judge；
- 分别评估引用精确率和引用召回率。

这次修复证明的是一套过程：**用指标发现问题 → 查看调用证据 → 提出多个方案 → 保护关键场景 → 增加测试 → 重跑评测 → 诚实说明边界**。
