# 迭代2详录（历史诊断）：把引用召回缺口变成可失败的测试

> 历史说明：这一轮的目标不是立即修改检索策略，而是证明迭代1的“引用准确率100%”没有覆盖引用召回，并把缺口写成能够稳定失败的评测用例。动态证据、Token预算和Precision/Recall/F1属于下一轮修复，完整时间线见[工程日志](engineering-log.md)。

这份记录展示如何用更强的评测集推翻一个看似全绿的中间结论：系统返回的引用都正确，不等于系统找全了回答所需的引用。

## 1. 扩展前的中间基线

迭代1结束时，10条固定用例和10项自动测试全部通过，报告中的“引用准确率”为100%。但这个指标实际只检查来源文件：

```text
引用准确率 = 实际来源中属于预期来源的数量 / 实际来源数量
```

评测用例只声明 `expected_sources`，例如期望 `release_policy.md`，没有声明必须引用该文件的哪些段落。回答器还固定最多选择两条证据：

```text
保留第一条证据；
第二条达到第一条分数的50%时保留；
最多两条。
```

这套设计能发现“夹带了错误来源”，却无法发现以下情况：

- 同一文件需要3段证据，实际只返回2段；
- 跨文件问题覆盖了两个来源，却漏掉其中一个问题要点；
- 系统正确拒答，但被通用的运行成功率记成失败。

因此，100%只证明实际返回的来源没有越界，不证明答案完整。

## 2. 新增用例怎样针对旧规则

评测集从10条扩展到15条。新增用例不是同类问题的简单增量，而是分别攻击旧评测的盲区：

| 用例 | 类型 | 设计目标 | 必须满足的证据或行为 |
|---|---|---|---|
| `case_11` | 同文档多段 | 超过固定两条上限 | 发布窗口p2、发布测试p3、日志期限p4 |
| `case_12` | 跨文档多段 | 检查跨来源且多要点的完整性 | 事故响应p2、紧急发布p2、发布测试p3 |
| `case_13` | 困难负样本 | 共享“生产、发布、Agent、测试”等词时仍要定位正确段落 | 发布测试p3 |
| `case_14` | 预期失败 | 区分正确拒答与运行故障 | `failure_reason=no_result` |
| `case_15` | 同文档多段 | 检查事件处理与日志字段是否都被回答 | 安全策略p3、p4 |

同时，正常用例增加精确到段落的 `expected_citations`；复合问题增加具名 `aspects`；预期拒答增加 `expected_failure`；用例再按 `single_evidence`、`multi_paragraph`、`multi_document`、`hard_negative` 和 `expected_failure` 分类。

例如，`case_11` 不再只声明来源文件，而是把完整性写入数据：

```json
{
  "id": "case_11",
  "category": "multi_paragraph",
  "expected_citations": [
    "release_policy.md#p2",
    "release_policy.md#p3",
    "release_policy.md#p4"
  ],
  "aspects": [
    {"name": "发布窗口"},
    {"name": "发布测试"},
    {"name": "日志期限"}
  ]
}
```

## 3. 用迭代1代码重放扩展集

复现口径是：使用迭代1结束提交 `a109ba5` 的Harness运行扩展后的15条用例。由于扩展评测和后续修复最终在同一次提交中落库，下面的数据是对历史代码的重放，不是单独提交过的报告产物。

旧选择策略在关键用例上的结果：

| 用例 | 预期引用 | 实际引用 | 旧指标 | 真正的问题 |
|---|---|---|---:|---|
| `case_11` | 发布p2、p3、p4 | 发布p3、p2 | 100% | Recall为66.67%，漏掉日志期限 |
| `case_12` | 事故p2、发布p2、p3 | 发布p2、事故p2 | 100% | Recall为66.67%，漏掉发布测试 |
| `case_13` | 发布p3 | 发布p3 | 100% | 困难负样本通过 |
| `case_14` | `no_result` | `no_result` | 0% | 行为正确，却被旧汇总当作失败 |
| `case_15` | 安全p3、p4 | 安全p3 | 100% | Recall为50%，漏掉越权处理 |

前三条不完整回答中的每一条实际引用都属于正确来源，所以旧的来源级“引用准确率”仍然是100%。这正是Precision与Recall不能互相替代的原因。

## 4. 用精确引用重新计算

排除预期拒答，14条可回答用例一共要求20条段落级引用。旧方案实际返回17条，17条都正确，但缺少3条：

```text
Micro Precision = 17 / 17 = 100%
Micro Recall    = 17 / 20 = 85%
```

按当前报告采用的逐题平均口径重新计算：

| 指标 | 旧方案在扩展集上的真实表现 |
|---|---:|
| Citation Precision | 100% |
| Citation Recall | 91.67% |
| Citation F1 | 94.76% |
| 精确引用集合匹配率 | 78.57%（11/14） |
| 按完整预期行为计算的任务通过率 | 80%（12/15） |

旧报告直接汇总15条用例时，`success_rate`、`retrieval_hit_rate` 和 `citation_accuracy` 都显示93.33%。下降并不是因为检索退化，而是 `case_14` 的正确 `no_result` 被统一计为0。这暴露了另一处指标定义错误：运行没有返回答案，不一定代表任务失败。

## 5. 方案比较

| 方案 | 能解决什么 | 仍然存在的问题 | 结论 |
|---|---|---|---|
| 把固定上限从2改成3 | 可让当前两个三证据问题多取一条 | 遇到4条证据仍失败，也可能重新引入弱引用 | 放弃 |
| 继续只检查关键词 | 能发现部分答案缺词 | 无法定位缺失的是哪段证据，通用词也可能误通过 | 放弃 |
| 把 `no_result` 改写成运行成功 | 汇总数字看起来一致 | 混淆流程状态与任务预期 | 放弃 |
| 增加精确引用、要点和预期失败 | 让少引用、漏要点和正确拒答都可独立验证 | 需要升级报告Schema | 采用 |

这一轮的关键选择是先修评测合同，再修检索器。如果直接把证据上限改成3，公开用例可能再次全绿，但“答案是否完整”仍然没有可持续验证的定义。

## 6. 交给下一轮的失败合同

扩展集要求后续实现同时满足：

1. `case_11` 返回3段同文档证据；
2. `case_12` 返回3段跨文档证据；
3. `case_15` 同时覆盖越权处理和日志字段；
4. `case_13` 不因大量共享词夹带错误段落；
5. `case_14` 返回 `no_result`，同时被计为任务通过；
6. 报告分别输出Precision、Recall、F1、精确集合匹配率和要点覆盖率；
7. `task_pass_rate` 与 `runtime_success_rate` 分开计算。

这些条件在下一轮推动了动态证据数量、BM25、复合问题分解、Token预算和评测Schema 2.0。修复后的15条用例在Precision、Recall、F1、精确引用匹配率、要点覆盖率和任务通过率上均达到100%；`runtime_success_rate` 保持93.33%，因为正确拒答不应伪装成成功回答。

## 7. 这一轮的边界

这轮没有证明检索器已经解决多证据问题，只证明旧的100%结论不完整，并把三个召回缺口和一个预期失败语义写进了回归测试。15条公开虚构用例仍然很小，而且参与了策略设计；它们适合作为确定性回归基线，不代表生产知识库上的泛化效果。

这一轮留下的工程结论是：**先让缺陷能够稳定失败，再讨论怎样让实现通过；只度量“引用有没有错”，会系统性忽略“必要引用有没有找全”。**
