# 工程日志：从可运行 Demo 到可回归基线

这份日志按时间保留每轮迭代的触发问题、可观察证据、方案选择、代码变化、回归结果和剩余边界。

## 迭代0：先建立确定性基线

### 观察到的问题

普通 RAG Demo 可以展示一次回答，却不能稳定回答：是否找对证据、失败在哪一步、修改后是否真的改善。

### 做出的选择

- 使用虚构 NovaLab 文档，避免暴露真实资料；
- 使用本地确定性检索和抽取式回答，不让外部模型随机性干扰基线；
- 建立10条固定问题、JSONL轨迹、失败分类和8项自动测试；
- 保留Token和费用字段，为以后接入真实模型预留接口。

### 得到的基线

检索命中率100%，关键词通过率100%，引用准确率90%。这说明“至少找到了正确文档”，但两条用例夹带了额外来源。

## 迭代1：减少弱相关引用

详细过程见[迭代1详录（历史方案）](citation-accuracy-fix.md)。

### 触发证据

- `case_03` 把只共享“生产”的安全策略加入发布时间回答；
- `case_07` 把只共享“分钟”的事故响应加入网关限流回答。

### 当时的改动

第二条证据只有达到第一条得分的50%才会进入回答，同时保留真正需要双来源的 `case_05`。自动测试从8项增加到10项，当前10条用例上的引用准确率从90%提升到100%。

### 后续发现的缺陷

规则仍固定最多两条证据。它提高了引用精确率，却可能隐藏多证据问题的引用召回损失；原评测只计算“实际引用里有多少正确”，没有检查“必要引用是否找全”。

这轮结果因此被保留为中间阶段，而不是最终方案。

## 迭代2：把召回缺口变成可失败的测试

详细过程见[迭代2详录（历史诊断）](citation-recall-gap.md)。

### 新增用例

评测集从10条扩展到15条：

- `case_11`：同一文档的3段证据——窗口、测试、日志期限；
- `case_12`：跨文档的3段证据——P0响应、紧急批准、发布测试；
- `case_13`：包含大量相似词的困难负样本；
- `case_14`：明确期待 `no_result`；
- `case_15`：越权处理和日志字段两个要点。

每条正常用例新增精确到段落的 `expected_citations`，复合问题新增具名 `aspects`。

### 旧方案在扩展集上的表现

| 用例 | 预期 | 实际 | 暴露的问题 |
|---|---|---|---|
| `case_11` | p2、p3、p4 | p3、p2 | 漏掉日志期限 |
| `case_12` | 事故p2、发布p2、发布p3 | 发布p2、事故p2 | 漏掉发布测试 |
| `case_15` | 安全p3、p4 | 安全p3 | 漏掉越权处理 |
| `case_14` | `no_result` | `no_result` | 行为正确却被旧成功率计为失败 |

旧报告仍会给前三条“引用准确率100%”，因为它没有引用Recall。这是指标定义而不是检索器单点问题。

## 迭代3：动态证据与评测Schema 2.0

### 评测指标纠错

将单一“引用准确率”拆为：

```text
Precision = 正确引用 / 实际引用
Recall    = 正确引用 / 预期引用
F1        = Precision与Recall的调和平均
```

新增：

- 精确引用集合匹配率；
- 问题要点覆盖率；
- 预期失败通过率；
- `task_pass_rate` 与 `runtime_success_rate` 分离；
- 按 `single_evidence`、`multi_paragraph`、`multi_document`、`hard_negative`、`expected_failure` 分类汇总。

### 检索和证据选择纠错

1. 将未归一化关键词得分替换为BM25，使用词频、逆文档频率和文档长度归一化；
2. 全局候选保留相对第一名达到45%的强匹配；
3. 对明确包含多个部分的问题进行保守分解，为每个子问题补充最佳证据；
4. 对“发生越权时……”这类事件表达抽取事件核心，避免通用“工具调用”掩盖稀有“越权”；
5. 取消固定证据条数，改用默认800近似Token的证据预算；
6. 标题段落不作为事实证据；
7. `evidence_selected` 轨迹记录每条候选的选择或拒绝原因。

### 可靠性补强

- 评测运行前重置对应轨迹，避免多次运行混在同一JSONL；
- 工具名称可从CLI传入，越权请求能端到端生成 `tool_boundary` 轨迹；
- Checkpoint指纹包含选择策略、阈值、Token预算和工具名，策略变化后不会误用旧结果；
- `--fail-under` 让评测低于门槛时返回非零退出码；
- GitHub Actions在Push和PR上自动运行测试与固定评测。

## 当前结果

| 指标 | 扩展前中间版 | Schema 2.0 当前版 |
|---|---:|---:|
| 固定用例 | 10 | 15 |
| 自动测试 | 10 | 16 |
| 引用Precision | 未独立报告 | 100% |
| 引用Recall | 未报告 | 100% |
| 引用F1 | 未报告 | 100% |
| 精确引用匹配率 | 未报告 | 100% |
| 要点覆盖率 | 未报告 | 100% |
| 预期失败通过率 | 未正确处理 | 100% |
| 任务通过率 | 无此指标 | 100% |

`runtime_success_rate=93.33%` 是预期的：15条中有1条要求失败为 `no_result`。不能把它改写成100%，否则会混淆运行成功与任务正确。

## 当前仍未解决

- 15条公开用例规模很小，且参与了当前规则验证；
- 复合问题分解依赖显式标点和轻量中文规则；
- 没有语义向量检索、Cross-Encoder或真实LLM；
- 证据预算使用近似Token；
- 工具安全尚未覆盖参数级权限、身份和沙箱；
- 还缺少未参与策略设计的独立盲测集。

下一轮最有价值的工作不是继续追求公开用例上的更高数字，而是增加单独的盲测文件、参数化对比和真实模型适配器。
